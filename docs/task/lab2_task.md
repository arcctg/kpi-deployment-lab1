# Завдання

## Опис завдання

Лабораторна робота складається з двох основних частин: дослідницької та практичної. Дослідницька частина полягає в тому, щоб дослідити різні підходи контейнеризації, виявити слабкі та сильні сторони кожного з підходів. Артефактом дослідницької частини має бути звіт, який детально описує виконані дослідження, спосіб їх відтворення, їх результати та зроблені висновки.

Практична частина полягає у практичному застосуванні отриманих знань та навичок для ефективної контейнеризації сервісів, необхідних для роботи застосунку із Лабораторної роботи №1. Артефаком практичної частини має бути файл docker-compose.yml в репозиторії з виконанням першої лабораторної роботи. Також студенти мають доповнити звіт дослідницької частини розділом, який описує які кроки було виконано для контейнеризації набору сервісів.

Після цього, вносимо зміни у файл [build/index.html](https://github.com/comsys-kpi-ua/deploy.lab-containers-starter-project-python/blob/main/build/index.html) або [spaceship/app.py](http://app.py) (додаємо коментар, заставляємо програму вивести ваші ім’я та прізвище, абощо) і повторно збираємо образ із новою версією коду.    `}`

   Порівняйте час збірки та розміри нового образу із numpy для образів на основі alpine та debian.

### Musl (Alpine) vs glibc (Debian/Ubuntu/etc)

1. Створюємо окрему мережу в docker для тестів:  
   `$ docker network create dns-lab`  
2. Запускаємо простий dns-сервер, який вміє резолвити наш кастомний домен  
   `$ docker run –rm -it --name dns-server --network dns-lab \`  
     `alpine sh -c "apk add dnsmasq && \`  
     `echo 'address=/myservice.internal.corp/10.0.0.50' > /etc/dnsmasq.conf && \`  
     `dnsmasq -k --log-queries --log-facility=-"`  
3. В новому вікні запускаємо контейнер з ubuntu, в якому пробуємо розрезолвити наш домен  
   `$ docker run --rm --network dns-lab \`  
     `--dns=$(docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' dns-server) \`  
     `--dns-search="corp" \`  
     `ubuntu:latest getent hosts myservice.internal`  
4. В новому вікні запускаємо контейнер з alpine  
   `$ docker run --rm --network dns-lab \`  
     `--dns=$(docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' dns-server) \`  
     `--dns-search="corp" \`  
     `alpine:latest getent hosts myservice.internal`  
5. Порівняти результати кроків 3 і 4, проаналізувати логи DNS сервера із п.1. та пояснити отримані результати. У висновках зробити припущення, до чого така поведінка системи може призводити.

## Практична частина

За допомогою [Docker Compose](https://docs.docker.com/compose/) автоматизувати запуск усіх 3 сервісів (web application, nginx, database) із [Лабораторної роботи №1](https://docs.google.com/document/u/0/d/1he74GoGW2TG7JJEJ0ELwcH_XAZ5yCkig3uGwTrbfEeg/edit). Сервіси повинні підніматися в окремій мережі (не в мережі за замовчуванням), дані БД повинні зберігатися на диску і переживати перезапуск контейнерів, видалення docker, перезавантаження системи.

## Git репозиторій

Код ваших Dockerfile має бути розміщений на GitHub. Файл для Docker Compose має знаходитися в репозиторії лабораторної роботи №1. Репозиторій лабораторної роботи №1 доповнити документацією по запуску проєкту за допомогою Docker Compose.

Історія комітів повинна бути логічною та зрозумілою, відображати хід виконання роботи студентом.

## Звіт

На основі проведених експериментів із Python-застосунком, Golang-застосунком та застосунком обраною вами мовою підготувати звіт. У звіті описати:

- що ви робили  
- які результати (розміри/час збірки/тощо) отримали  
- з якими особливостями/труднощами стикалися, які їх об’єктивні причини  
- ваші висновки по отриманих результатах. На основі ваших висновків напишіть ваші рекомендації стосовно упаковки застосунків у контейнери.

Звіт ви готуєте у Google Docs / Word / або у вигляді веб\-сторінки і завантажуєте документ або посилання на ваш звіт для здачі лабораторної роботи. У звіті ви не повинні дублювати прям весь код, натомість ви вказуєте тільки найважливіші фрагменти та робите посилання на конкретні файли у ваших репозиторіях, щоб можна було дізнатися більше. 

Експерименти у звіті повинні бути відтворювані. Ви маєте вказати, як відтворити отримані вами результати (наприклад, цей експеримент запускався на такій-то конфігурації. Версія коду — така-то (лінк), осьо докерфайл (лінк), осьо команда для збору образу. 

**Важливо:** не лити воду у звіті. Давайте збережемо один одному час і обійдемося без шаблонних “на цій лабораторній роботі я навчився(-лася) збирати docker-образи” і подібних фраз, які зазвичай просять вставити у звіт заради формальностей. Ви провели цілу низку експериментів. То розкажіть про них\!
